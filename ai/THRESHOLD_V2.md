## 1. Step 1 — Label Noise Parameters

The original `dynamic_t_block` / `dynamic_t_review` values were generated by
a deterministic formula: **t_block = clamp(75 − (risk−50)×0.5, 40, 95)** and
**t_review = t_block − 25** (constant gap of 25 points). The v1 XGBoost classifiers
were trained to reproduce this formula's *output* (verdict class), so they were
circular — the model just re-derived arithmetic.

To make the regression target non-trivial, independent Gaussian noise was added
in `scripts/12_add_threshold_noise.py`:

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| RNG seed | 42 | Fixed for reproducibility |
| σ(t_block) | **4.0 points** | ≈ 56% of the natural σ of t_block (~7.2), adding meaningful variance without swamping the risk-score trend |
| σ(t_review) | **3.5 points** | Slightly smaller so block and review targets carry independent information |
| Outer clamp | t_block ∈ [10, 95], t_review ∈ [5, 95] | Prevents nonsensical values; no tight inner clamps |
| Ordering fix | if t_review ≥ t_block → t_review = t_block − 1 | Safety only; triggered 0 times on this dataset |

**Before vs after noise:**

| Metric | t_block (before) | t_block (after) | t_review (before) | t_review (after) |
|--------|-----------------|-----------------|-------------------|------------------|
| Mean | 88.154 | 87.772 | 63.154 | 63.123 |
| Std  | 7.172  | 7.751  | 7.172  | 7.959  |
| Min  | 52.521 | 41.340  | 27.521 | 17.675  |
| Max  | 95.000 | 95.000  | 70.000 | 86.711  |
| Gap (t_block − t_review) std | 0.00002 (constant) | **4.958** | — | — |

## 2. Step 2 — XGBoost Threshold Regressor

Two separate `XGBRegressor` models (one per target) trained in
`scripts/13_train_threshold_regressor.py`.

**Why two separate models instead of XGBoost multi-output?**  
Portability (XGBoost 1.x and 2.x), separate per-target feature importances
for cleaner audit narrative, and no API instability from a recently-added
experimental feature.

**Feature set (26 features):**

*Account-level (18):* overall_risk_score, geographic_risk, identity_kyc_risk,
pep_sanctions_risk, behavioural_risk, relationship_network_risk, kyc_completeness,
kyc_status_enc, account_type_enc, is_pep, has_complex_ownership, shell_company_flag,
activity_tier_enc, account_status_enc, shares_address_with_sanctioned,
pep_exposure_score, country_risk_score, override_applied

*Transaction-level (8, zero-filled for account-context rows):* has_transaction,
amount_log, payment_rail_enc, is_first_time_recipient, velocity_30d_count,
velocity_30d_amount_log, hour_of_day, day_of_week

*Excluded:* hops_to_sanctioned, name_match_type_enc (label-proxy / leakage risk),
match_score (compared to the threshold, not a driver of it), dynamic_verdict,
verdicts_differ (derived from targets).

## 3. Step 3 — `compute_thresholds()` / `screen()` Changes

### compute_thresholds(context) → dict

```python
thresh = compute_thresholds(context)
# thresh["min_block"]  — predicted block threshold
# thresh["min_review"] — predicted review threshold
# thresh["reasons"]    — list[str] importance-weighted explanations
```

**Algorithm:**
1. Encode `context` dict into 26-feature float32 vector
2. Run `block_model.predict(x)` → raw_block
3. Run `review_model.predict(x)` → raw_review
4. Apply outer clamps: t_block = clip(raw_block, 10, 95); t_review = clip(raw_review, 5, 95)
5. Ordering safety: if t_review ≥ t_block → t_review = t_block − 1
6. Build `reasons` from importance-weighted contribution proxy (not SHAP)

**Post-prediction clamps:**

| Threshold | Min | Max | Notes |
|-----------|-----|-----|-------|
| t_block   | 10  | 95  | Outer safety only — no tight inner clamp |
| t_review  | 5   | 95  | Outer safety only — gap to t_block varies freely |

The gap between t_block and t_review is now **variable** (std ≈ 5 points in the
training data, mean ≈ 25) rather than the rigid constant 25 of the v1 formula.

### screen() changes

- Now calls `compute_thresholds()` for threshold prediction (replaces the hardcoded formula)
- Returns two new fields: `min_block` / `min_review` (same values as `t_block` / `t_review`
  which are kept as aliases for backward compatibility)
- Returns `threshold_reasons: list[str]` — new field surfacing the threshold explanation
- `audit_factors` now includes the top-2 threshold reason strings
- v1 classifiers still loaded and used for `block_probability` / `class_probabilities`
  (marked DEPRECATED in model_metadata.json)

## 6. What This Model Is and Isn't

This v2 threshold model is a **learned approximation of a noisy version of
the original deterministic formula**, trained on synthetic data. It is designed
for use in the **Phase-0 / shadow-mode** described in the original proposal:

**What it is:**
- A working XGBoost regression architecture that predicts `(t_block, t_review)`
  as a continuous pair rather than deriving them from a single formula
- A demonstration that the threshold prediction task is non-trivial when labels
  have realistic variance (R² ≈ 0.80, not 1.00)
- Ready for retraining on real analyst feedback without architectural change —
  just swap in real `(match_score, decided_threshold)` labels as they accumulate
- A richer feature set than the v1 formula (transaction amount, payment rail,
  velocity, first-time recipient, time-of-day) that would be genuinely predictive
  once trained on real human-annotated thresholds

**What it isn't:**
- Trained on real analyst decisions — it learned a noisy version of the formula,
  not human judgment
- A SHAP-based explainer — `threshold_reasons` uses importance-weighted
  contribution as a fast proxy; for true SHAP use `shap.TreeExplainer`
- A replacement for compliance policy — thresholds should be audited against
  regulatory requirements before production use

**Retraining path:** As real threshold decisions accumulate (analyst overrides,
regulatory feedback, case outcomes), replace `dynamic_t_block` / `dynamic_t_review`
labels with those real values and re-run `scripts/13_train_threshold_regressor.py`.
No architectural change required.
